# bip-39-japanese-for-kanji（漢字混じりメモ風形式の漢字ワードリスト）設計

**Goal:** BIP-39 日本語（ひらがな2048語）を **index 完全一致のまま漢字へ"スキン"**した 2048 語リストを整備し、Issue #2 の「漢字混じりメモ風形式」を表記揺れリスクを抑えた形で実装可能にする。

**親 Issue:** #2（保存形式を追加：目立ちにくい形式 / 漢字混じりメモ風）。本 spec はそのうち**漢字ワードリスト**部分のみを対象とする（「目立ちにくい形式＝payloadのみ」は別作業）。

**Status:** ドラフト（本形式は #2 方針どおり**実験扱い**・長期保存は標準形式推奨）。

---

## 1. 背景と方針

- 既存のテキスト表現は 2 つ：標準形式（`OSM1.`+base64url）と日本語単語列形式（BIP-39 ひらがな2048語・`、`区切り・11bit base-2048）。
- #2 が漢字形式を「実験扱い」とした最大の理由は **標準化された漢字2048語リストが無く、表記揺れ（異体字・送り仮名・同音異義の書き分け）で復号が壊れるリスク**。
- そこで「新規に漢字語を選ぶ」のではなく、**BIP-39 日本語を漢字化（index 一致の漢字スキン）**する。出自・ライセンス・語数・順序を BIP-39 から継承でき、表記揺れは"変換規則"で封じ込める。
- 安全に漢字化できない語は**ひらがなのまま残す**＝結果は文字どおり「漢字混じり」（#2 の名称と一致）。

### 確定した 3 つの決定（brainstorm 結果）
1. **復号モデル**：既存の 11bit base-2048 コーデックを流用。`、`(U+3001) 区切り、4バイトBE長さプレフィックス、チェックサムワードなし。`WORDLIST` を差し替える構造。
2. **語の出自**：BIP-39 日本語（`japanese.txt`）を漢字化（新規選定ではない）。
3. **変換モデル**：**index 完全一致の漢字スキン**。安全に漢字化できる語のみ漢字、他はかな維持。同じ payload は標準/ひらがな版/漢字版で**同じ index**に符号化され、表記だけ異なる。

---

## 2. ワードリストの条件（=本 spec の核心）

### 2.1 構造・互換性
- ちょうど **2048 語**、BIP-39 日本語と**同一順序・同一 index**（`位置 i = BIP-39 語 i のスキン`）。
- 凍結データ：`crates/core/data/bip39-japanese-kanji.txt`（2048 行・改行区切り）。ビルド時に配列化（既存 `wordlist_array.in` と同方式で `wordlist_kanji_array.in` を生成）。
- 出自・ライセンス：bitcoin/bips `japanese.txt` の派生物である旨と継承ライセンスをデータファイル先頭コメントおよび SPEC に明記。

### 2.2 漢字化を「採用」する条件（**すべて**満たす語だけ漢字化）
- (a) 使用する漢字がすべて **常用漢字**（人名用・表外字・旧字体・異体字は不可）。
  - ※既定はこの厳しさ。緩和（常用＋人名用）は §4 の実測チェックで判断する。
- (b) **漢字部分がすべて常用漢字**なら、送り仮名・かな接尾辞を含む標準表記も可（`赤ちゃん`/`明け方`/`編み物`/`楽しみ`）。語は符号表のシンボルであり「純漢字」である必要はない。
- (c) **候補の選定**（同音の複数表記の解決）：(i) 候補が1つならそれを採用（`愛国心` 等の頻度タグ無し複合語も拾う）、(ii) 複数なら JMdict 頻度マーカー(`ke_pri`)付きの最頻形を採用（`感謝` > `官舎`）、(iii) 複数候補で頻度マーカーが皆無なら obscure な当て字（`亜米利加` 等）とみなしかな維持。
- (d) 標準的正書法（常用漢字表の音訓・一般的熟語）で辞書確認できること。NFC 安定であること。

### 2.3 漢字以外の選択（→ 結果は「漢字・カタカナ・ひらがな混じり」）
§2.2 の漢字に該当しない語は、次の優先順で自然な標準表記を選ぶ:
- **外来語・外国語**（JMdict の読みがすべてカタカナの語）は**カタカナ**で採用：`あめりか`→`アメリカ`、`たいみんぐ`→`タイミング`。
- **固有名詞**（地名・人名）は **JMnedict** に常用漢字の単一表記があれば**漢字**：`かなざわし`→`金沢市`。
- いずれにも当てはまらない語（かな書きが普通の和語＝`うっかり`、obscure な当て字のみ、衝突回避）は **BIP-39 のひらがなのまま**。

### 2.4 一意性・復号安全性（**最重要**）
- **変換後 2048 語が全て相異なる文字列**であること。漢字化で衝突（別の 2 語が同一漢字表記）が出たら、衝突側を**かな維持に倒して**回避。
- **正規化**：BIP-39 日本語リストは **NFKD（濁点等が分解形）**。したがって — (1) 辞書照合は NFC 正規化して行う、(2) **かな維持エントリは index 一致のため BIP-39 の正準バイト列（NFKD）をそのまま継承**（既存ひらがな形式と同一の正規化前提）、(3) **漢字エントリは NFC 安定な常用漢字のみ**（CJK 互換漢字・IVS を含まない＝あらゆる正規化形で不変）。変換後 2048 語が相異なること。
- 漢字表は「各 index につき **{漢字 / カタカナ / かな} の 1 表記のみ**を持つ単一表」。
- 自動判定 `detect_and_decode` の順序：**(1) 標準 →(2) ひらがな版 →(3) 漢字版**で「全トークン一致」を試す。
  - 漢字トークンを 1 つでも含む → ひらがな版照合は失敗 → 漢字版で成功（**判別可能**）。
  - 全トークンがかな維持語のみ → 両版で成功するが **index 一致のため同一 payload に復号**（実害なし）。
- トークン分割は `、` 固定、各トークン trim・空要素無視（既存 `decode_words` 踏襲）。

### 2.5 可搬性・レンダリング（"いつでも復号できる"の担保）
- 全漢字が常用漢字＝主要 OS / フォント（**recovery kit のオフライン環境を含む**）で確実に表示・入力可能。
- **1 コードポイント＝1 通常漢字のみ**。絵文字・環境依存文字・**IVS / 異体字セレクタ・CJK 互換漢字・結合文字は禁止**（NFKC 不変と整合）。

### 2.6 凍結・バージョニング
- **一度リリースしたら永久凍結**（1 文字でも変えると過去テキストが復号不能）。
- 誤り発覚時は「修正版＝**別フォーマット種別 / 別データファイル**」として新規追加（既存表は不変のまま）。

## 3. フォーマット統合（実装の外形）

- 形式種別 `"kanji"` を `encode`/`decode`/`detect_and_decode` と wasm `encrypt`（既存の format 厳格化）に追加。
- エンコード規則は既存 `encode_words` と**完全同一**（表だけ差し替え）。`KANJI_WORDLIST: [&str; 2048]` を追加し、`encode_words`/`decode_words` を表パラメータ化するか、漢字版関数を薄く追加。
- 不正語は既存 `error_kind: "invalid_word"` を流用（新 error_kind は追加しない）。
- UI に保存形式として追加。**実験ラベル＋「長期保存は標準形式推奨」注記**を付す（#2 方針どおり、今時点では実験扱い）。

---

## 4. 生成・検証パイプライン（"やってみて確認"を内蔵）

1. **機械生成**：各 BIP-39 語について、辞書ソースから漢字候補を引き、§2.2 の安全フィルタ（常用漢字のみ・送り仮名なし・同音異義非分裂・正書法）を適用。落ちた語はかな維持。
2. **実測レポート（確認ゲート）**：`漢字化率`（2048 中いくつ漢字化したか）と**かな維持語の全リスト**、衝突回避でかなに倒した語を出力。
   - ここで実数を見て、**常用漢字のみで十分か / 常用＋人名用へ緩めるか**を判断する（Q-a「やってみて確認したい」の具体化）。
3. **人手監査**：全 2048 を目視確認（同音異義の誤選択・常用外混入・不自然な熟語の検出）。
4. **凍結**：最終確定値を `bip39-japanese-kanji.txt` に固定。変換スクリプトと根拠（常用漢字表・辞書ソース）はリポジトリに残すが、**確定値は txt が正**（将来のスクリプト挙動変化に非依存）。

### データソース / ライセンス
- 漢字対応の根拠（辞書・常用漢字表）は出典を記録。BIP-39 `japanese.txt` 由来である旨とライセンスを継承表記。
- データソースの選定基準（実装計画で最終確定）：**(1) 権威ある常用漢字集合**（内閣告示「常用漢字表」由来）で使用文字を判定、**(2) 機械可読な公開辞書**（例: JMdict/EDICT 系など、ライセンス互換を確認の上）で各 BIP-39 語の漢字候補を取得。いずれも**ライセンス互換性を確認**してから採用し、不採用時はかな維持にフォールバックする（コードパスは辞書ソースに依存しない）。

---

## 5. テスト（受け入れ条件）
- `bip39-japanese-kanji.txt` が **2048 行**。
- **全語ユニーク**（変換後）。
- **全語 NFKC 安定**、かつ NFKC 後も全ユニーク。
- 使用文字が**すべて常用漢字**（人名用・表外字・異体字・IVS を含まない）※緩和を採った場合はその集合に追従。
- **ひらがな版と index 整合**（行数・対応位置が一致）。
- 漢字版 **ラウンドトリップ**（encode→decode で payload 一致、様々な長さ）。
- `detect_and_decode` が漢字版テキストを正しく判別（漢字トークンを含むケース）。
- 漢字版**テストベクタ**を `spec/test-vector.json`（または専用ファイル）に追加し SPEC に明記。

---

## 6. スコープ外 / YAGNI
- 「目立ちにくい形式（payloadのみ・プレフィックスなし）」＝ #2 の別項目、本 spec 対象外。
- 区切りなしの文章風エンコード／助詞風トークン挿入（一意復号が脆くなるため不採用）。
- チェックサムワード（誤記検出）＝別 Issue #3。
- 文章としての文法的自然さ（主語・述語のスロット化）＝本形式は「漢字混じり語列」までに留める。

---

## 7. リスク・注意
- 表記揺れは変換規則で大幅低減するが、**漢字は環境依存・コピー経路での化けがひらがなよりわずかに起こりやすい**。標準形式が常に第一推奨である点は UI / SPEC で明示。
- 実測の漢字化率は約 **87%**（頻度ベース選定＋かな混じり許可）。残りは固有名詞・obscure 当て字・衝突回避でかな維持。「ほぼひらがな」化の懸念は解消。
- 凍結後は不可逆。リリース前の人手監査を必須とする。
